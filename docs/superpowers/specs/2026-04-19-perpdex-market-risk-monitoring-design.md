# PerpDEX 异常市场结构监控系统 Spec

- 日期: 2026-04-19
- 状态: Draft v1
- 文档类型: Technical Spec
- 目标读者: 后端工程、数据工程、量化研究、风控、产品

## 1. 背景

部分小市值代币在链上永续合约平台上可能出现一种异常市场结构：

1. 现货流通盘高度集中，少量地址或关联实体可显著影响现货价格。
2. 永续合约市场的未平仓合约量（Open Interest, OI）相对现货成交量异常偏大。
3. 平台流动性较薄或订单簿深度不足，较小资金即可推动价格偏离。
4. 合约标记价格、指数价格依赖现货报价或跨平台价格输入，现货扰动可被放大传导至合约市场。
5. 跨所套利机器人、做市商与被动流动性可能在价格偏离后被吸引入场，成为头寸转移的对手盘。

本文档定义一个“异常市场结构监控系统”，用于持续识别上述风险信号，并为后续人工研究、风控拦截或报警系统提供依据。

该系统用于风险观察、异常提示和研究分析，不用于执行交易或辅助操纵市场。

## 2. 问题定义

我们希望回答以下问题：

1. 哪些币种正在出现与自然交易行为不匹配的 OI、资金费率、价差和流动性结构？
2. 哪些币种同时满足“高控盘现货 + 高 OI + 低自然换手 + 跨所价差异常”的组合特征？
3. 哪些异常是短时噪音，哪些异常具备持续性、可升级为重点跟踪对象？
4. 哪些异常在多个平台之间发生传导，形成更高风险的结构性事件？

## 3. 目标

系统目标如下：

1. 对指定币种集合持续采集 CEX、PerpDEX 与链上公开数据。
2. 计算一组面向异常市场结构的核心监控指标。
3. 以规则引擎和风险评分方式，对币种进行分层告警。
4. 记录时序数据与事件快照，支持回看、复盘和策略迭代。
5. 为后续开发完整监控平台提供数据模型、模块边界和 MVP 路线。

## 4. 非目标

以下内容不属于本期系统范围：

1. 不判定某地址或机构存在违法操纵行为。
2. 不尝试识别真实身份或做归因执法。
3. 不直接下单、对冲、套利或生成自动交易建议。
4. 不依赖私有交易所数据、KYC 数据或非公开订单流。
5. 不在 v1 中构建复杂机器学习模型，优先使用可解释规则和统计阈值。

## 5. 核心监控假设

系统建立在以下可观测假设上：

### 5.1 现货控盘假设

如果一个币种的流通筹码高度集中，则少量资本即可在低流动性的现货市场中显著影响成交价格。

可观测替代信号：

1. 前 N 地址持仓占比高。
2. 关联地址簇持仓占比高。
3. 活跃自由流通量低。
4. 链上大额转账与价格异动高度同步。

### 5.2 合约放大假设

如果合约市场 OI 显著大于自然成交需求，则现货价格扰动可在永续市场形成放大的账面盈亏与清算压力。

可观测替代信号：

1. OI / 24h 现货成交量异常高。
2. OI / 自由流通市值异常高。
3. OI 在短时间内阶跃式增长，但成交量未同步增长。
4. 清算事件与价格波动之间呈非自然不对称。

### 5.3 价格传导假设

如果平台指数价格或市场预期受现货价格输入影响，则小市场上的价格冲击可以传导到更大市场。

可观测替代信号：

1. PerpDEX 与 CEX 现货或永续价格产生持续溢价或折价。
2. 偏离出现时，套利方向上的资金费率持续偏斜。
3. 偏离产生后，另一个平台的 OI 和成交量出现跟随增长。

### 5.4 对手盘吸引假设

当价格偏离足够明显时，套利机器人和做市商会主动进入，形成头寸转移机会。

可观测替代信号：

1. 某平台价差扩大后，另一平台短时成交量放大。
2. 价差回归前，偏离平台 OI 未下降反而继续扩张。
3. 资金费率长期维持单侧极端水平，暗示大量同向拥挤仓位。

## 6. 监控对象

### 6.1 币种范围

系统支持三类监控名单：

1. 固定观察名单：研究员手工指定的高风险币种。
2. 动态发现名单：满足初筛规则的币种自动纳入观察。
3. 事件升级名单：近期多次触发高风险事件的币种。

### 6.2 平台范围

v1 建议覆盖以下数据域：

1. CEX 现货和永续：如 Binance。
2. PerpDEX：如 Aster，后续可扩展到 Hyperliquid、Aevo、dYdX 等。
3. 链上数据：代币持仓分布、转账、池子流动性、合约事件。
4. 行情聚合与辅助数据：如 Coingecko、CoinMarketCap、DexScreener、Coinglass 或自建抓取。

## 7. 核心指标体系

### 7.1 市场结构指标

1. `spot_24h_volume_usd`
2. `perp_24h_volume_usd`
3. `open_interest_usd`
4. `oi_to_spot_volume_ratio = open_interest_usd / max(spot_24h_volume_usd, epsilon)`
5. `oi_to_market_cap_ratio = open_interest_usd / circulating_market_cap_usd`
6. `oi_to_free_float_ratio = open_interest_usd / estimated_free_float_usd`
7. `perp_volume_to_oi_ratio = perp_24h_volume_usd / max(open_interest_usd, epsilon)`

### 7.2 价格偏离指标

1. `cex_spot_price`
2. `cex_perp_price`
3. `dex_perp_price`
4. `index_price`
5. `mark_price`
6. `premium_bps = (dex_perp_price - cex_reference_price) / cex_reference_price * 10000`
7. `premium_duration_sec`
8. `max_premium_last_1h`
9. `price_impact_estimate`，基于订单簿或 AMM 滑点估算。

### 7.3 资金与拥挤度指标

1. `funding_rate_hourly`
2. `annualized_funding_rate`
3. `long_short_ratio`，若可获取。
4. `liquidation_volume_long`
5. `liquidation_volume_short`
6. `funding_skew_duration`

### 7.4 流动性指标

1. `orderbook_depth_10bps_usd`
2. `orderbook_depth_50bps_usd`
3. `spread_bps`
4. `slippage_buy_10k_usd`
5. `slippage_sell_10k_usd`
6. `amm_pool_tvl_usd`，若适用。

### 7.5 链上集中度指标

1. `top_10_holder_ratio`
2. `top_20_holder_ratio`
3. `clustered_holder_ratio`
4. `estimated_free_float_ratio`
5. `new_wallet_share_change`
6. `large_transfer_count_1h`
7. `exchange_inflow_usd`
8. `exchange_outflow_usd`

### 7.6 异常变化指标

1. `delta_oi_5m`
2. `delta_oi_1h`
3. `delta_funding_1h`
4. `delta_premium_5m`
5. `delta_depth_15m`
6. `zscore_oi_to_volume_30d`
7. `zscore_premium_7d`
8. `zscore_funding_7d`

## 8. 风险模式定义

系统识别以下模式：

### 8.1 高控盘高 OI 模式

触发条件示例：

1. `top_20_holder_ratio >= 0.7`
2. `oi_to_spot_volume_ratio >= 3`
3. `perp_volume_to_oi_ratio <= 0.5`
4. 条件持续超过 30 分钟

风险含义：可能存在非自然高杠杆结构，现货扰动可对合约端产生显著影响。

### 8.2 价差诱导模式

触发条件示例：

1. `premium_bps >= 150`
2. `premium_duration_sec >= 300`
3. 偏离期内 `open_interest_usd` 持续增加
4. 另一主要平台成交量或 OI 同步抬升

风险含义：可能存在通过局部市场制造偏离、吸引对手盘入场的结构。

### 8.3 资金费拥挤模式

触发条件示例：

1. `annualized_funding_rate >= 100%`
2. 极端资金费率持续超过 6 小时
3. OI 未显著下降

风险含义：市场一侧拥挤，存在通过高位横盘或延迟回归消耗对手盘的条件。

### 8.4 流动性脆弱模式

触发条件示例：

1. `orderbook_depth_10bps_usd` 低于设定阈值
2. `spread_bps` 和 `slippage` 快速放大
3. 同时存在价格大幅波动与 OI 上升

风险含义：小额资金即可推动价格，市场更易出现非自然偏离和清算链条。

### 8.5 链上供给异动模式

触发条件示例：

1. 大额地址之间发生频繁转账
2. 转账后短时间内出现价格与 OI 异动
3. 交易所净流入突然增加

风险含义：筹码调度可能与市场操作时点相关，需要升级跟踪。

## 9. 风险评分模型

### 9.1 总体原则

v1 采用可解释加权评分，不使用黑盒模型。

总分范围为 0 到 100，分数越高代表异常市场结构风险越高。

### 9.2 建议分桶

1. 0 到 29: 低风险
2. 30 到 59: 观察
3. 60 到 79: 高风险
4. 80 到 100: 极高风险

### 9.3 建议权重

1. 现货集中度: 25
2. OI 与成交量结构异常: 25
3. 价差与价格传导异常: 20
4. 资金费率与拥挤度: 15
5. 流动性脆弱性: 10
6. 链上异动辅助项: 5

### 9.4 评分逻辑

每个维度包含若干标准化子项，采用 min-max 或分段函数映射到 0 到 100 后加权求和。

系统必须同时输出：

1. 总分
2. 各子维度分数
3. 触发的规则列表
4. 最近 1h 内的关键解释变量

## 10. 系统输入输出

### 10.1 输入

1. 币种列表与平台映射。
2. 平台 API 凭证或公开接口地址。
3. 规则阈值配置。
4. 链上地址标签或聚类规则。

### 10.2 输出

1. 实时风险评分。
2. 告警事件流。
3. 币种级别的时序指标快照。
4. 可供前端展示的聚合视图。
5. 可供研究回测的明细数据。

## 11. 功能需求

### 11.1 数据采集

系统应支持：

1. 定时拉取或订阅多平台行情、OI、资金费率、成交量和清算数据。
2. 定时拉取链上持仓分布、转账和流动性数据。
3. 对不同平台的 symbol、精度、时间戳与报价单位进行标准化。
4. 对临时数据缺失、延迟或平台故障进行容错处理。

### 11.2 指标计算

系统应支持：

1. 分钟级聚合计算核心指标。
2. 维护滚动窗口统计值，如 5m、15m、1h、24h、7d。
3. 计算 z-score、百分位和分段风险值。

### 11.3 规则引擎

系统应支持：

1. 通过配置文件定义阈值、窗口和持续时间。
2. 组合多条件规则生成事件。
3. 控制重复告警和冷却时间。
4. 支持按币种、平台、模式类型进行白名单或黑名单控制。

### 11.4 告警与通知

系统应支持：

1. 当风险评分跨桶时发送告警。
2. 当特定规则首次触发、持续触发或升级触发时发送告警。
3. 支持输出到日志、Webhook、Telegram、Slack 或邮件。
4. 每条告警附带解释字段，而不是只给单个分数。

### 11.5 研究与回看

系统应支持：

1. 查询某币种在指定时间区间的指标变化。
2. 查看某次告警触发前后的关键时间线。
3. 导出原始指标与事件样本，供后续研究和策略调优使用。

## 12. 非功能需求

1. v1 监控延迟目标小于 60 秒。
2. 单次指标计算失败不能阻塞整体管道。
3. 所有原始采集数据与衍生指标必须可追溯。
4. 规则与权重配置必须可热更新或低成本发布。
5. 关键模块应具备单元测试和回放测试能力。

## 13. 逻辑架构

系统划分为以下模块：

1. `connectors`: 平台与链上数据接入层。
2. `normalizer`: 统一 symbol、单位、时间戳和字段语义。
3. `feature-engine`: 指标和滚动窗口计算。
4. `risk-engine`: 规则判定和风险评分。
5. `event-store`: 原始数据、特征和告警事件存储。
6. `api-service`: 对前端和研究接口提供查询能力。
7. `alert-dispatcher`: 发送外部通知。

数据流如下：

1. 连接器采集原始数据。
2. 标准化层写入统一事件格式。
3. 特征引擎产出分钟级和窗口级指标。
4. 风险引擎基于指标触发规则并计算分数。
5. 事件写入存储，并由通知模块分发。

## 16. v1 实施范围

v1 建议只做最小闭环：

1. 先支持 1 个 CEX、1 个 PerpDEX、1 条链。
2. 先覆盖 10 到 50 个观察币种。
3. 先实现分钟级采集、规则引擎、评分、Webhook 告警。
4. 前端仅提供简单表格和详情页，复杂可视化后置。

## 17. 验证方法

### 17.1 历史回放验证

1. 选取已知异常币种样本。
2. 回放历史数据，验证系统是否在异常扩大前触发预警。
3. 记录召回率、提前量、误报率。

### 17.2 对照组验证

1. 选择一组正常流动性币种作为对照。
2. 验证在正常市场条件下告警频率是否可控。

### 17.3 研究员人工复核

1. 对高分事件进行人工标注。
2. 评估规则解释性和可操作性。
3. 持续调整阈值和权重。

## 18. 风险与限制

1. 公开数据存在缺失、延迟、口径差异和平台噪音。
2. 链上地址聚类存在误差，集中度估算并不精确。
3. 高风险信号不等于操纵事实，系统只能提示异常结构。
4. 某些平台的 OI、清算与资金费率口径可能变化，需要定期校准。
5. 规则阈值具有市场阶段依赖，不能静态假设长期有效。

## 19. 后续演进

后续可扩展方向：

1. 增加更多 PerpDEX 与链上数据源。
2. 引入关联地址聚类和实体画像。
3. 引入异常检测模型和事件相似度搜索。
4. 增加市场回放视图与事件注释功能。
5. 支持多租户和多策略配置。

## 20. 开发建议

建议按以下顺序开发：

1. 定义统一数据模型和市场字段字典。
2. 接入 Binance、Aster 与链上持仓分布数据。
3. 实现指标引擎和基础规则。
4. 实现风险评分和告警推送。
5. 实现查询 API 和最小前端面板。
6. 基于历史样本迭代阈值。

1. `open_interest_usd > 5_000_000`
2. `oi_to_spot_volume_ratio > 2.5`
3. `zscore_oi_to_volume_30d > 2`
4. 持续 15 分钟

### 规则 B: 持续高溢价

当满足以下条件时触发：

1. `premium_bps > 120`
2. `premium_duration_sec > 600`
3. `annualized_funding_rate > 80%`

### 规则 C: 高集中度脆弱市场

当满足以下条件时触发：

1. `top_20_holder_ratio > 0.75`
2. `orderbook_depth_10bps_usd < 100_000`
3. `spread_bps > 20`
